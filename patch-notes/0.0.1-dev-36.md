# 0.0.1-dev-36 — tuple destructure in `for-of` + tuple-array types

_Released 2026-05-27_

Two related fixes in one bump. `for (let [a, b] of pairs)`
now destructures each tuple element into a separate name,
and `[int, string][]` is now a valid type (it used to fail
at parse time because the array suffix `[]` only attached
to named / generic types).

## Highlights

```mint
import { println } from "mint:io";

function makePair(n: int, name: string): [int, string] {
  return (n, name);
}

export function main(): void {
  let p1: [int, string] = makePair(1, "alpha");
  let p2: [int, string] = makePair(2, "beta");
  let pairs: [int, string][] = [p1, p2];
  for (let [n, name] of pairs) {
    println(n.toString() + " " + name);
  }
}
```

Rules:

- Binding kind is `let` or `const`, same as the single-id
  form.
- Tuple arity must match the iterable's element tuple type
  (`DLM4086` for a mismatch).
- Iterable must be an `ArrayType<TupleType>` —
  destructuring a non-tuple iterable raises `DLM4085`.
- Lowered to C++ structured binding inside a range-based
  for: `for (const auto& [a, b] : pairs) { … }`.

## Files touched

- `src/parser/ast.ts` — `ForOfBinding` union with
  `style: "single" | "destructure"`.
- `src/parser/parser.ts` — `forStatement()` and
  `looksLikeForOf()` accept the `[a, b]` shape.
  `finishPostfixType()` now also greedily consumes `[]`,
  so `[int, string][]` parses correctly.
- `src/semantic/checker.ts` — destructure binding path
  resolves the iterable's tuple element type and emits
  `DLM4085` / `DLM4086`.
- `src/emitter/cppEmitter.ts` — `emitForOfStatement` picks
  the binding shape and emits the C++ structured binding
  when destructuring.
- `tests/integration/forOfDestructureBuild.test.ts` —
  happy path + arity-mismatch diagnostic.
