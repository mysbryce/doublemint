# 0.0.1-dev-31 — compound `<<= ` / `>>=`

_Released 2026-05-27_

Closes the operator family. Compound shifts piggyback on
the lexer pair `(LESS, LESS_EQUAL)` and `(GREATER,
GREATER_EQUAL)` — those two are the exact tokenisation
of `<<=` / `>>=` — and the parser fuses them at the
`assignment()` layer.

## Highlights

```mint
import { println } from "mint:io";

export function main(): void {
  let n: int = 1;
  n <<= 4;
  if (n == 16) { println("shl-eq ok"); }

  let m: int = 256;
  m >>= 2;
  if (m == 64) { println("shr-eq ok"); }
}
```

Rules:

- LHS must be a mutable assignment target (shared with the
  rest of the assignment family).
- Both operands must be integer (`int` / `int64` / numeric
  literal); `DLM4082` otherwise.

## Files touched

- `src/parser/ast.ts` — `AssignmentExpression.operator`
  widened with `"<<=" | ">>="`.
- `src/parser/parser.ts`:
  - `assignment()` detects `LESS LESS_EQUAL` and
    `GREATER GREATER_EQUAL` pairs and consumes them as the
    new compound operators.
  - `comparison()` no longer matches a bare `<` / `>` when
    the next token is `LESS` / `LESS_EQUAL` (or
    `GREATER` / `GREATER_EQUAL`), so it doesn't steal
    bytes meant for `shift()` or `<<=` / `>>=`.
- `src/semantic/checker.ts` — `DLM4082`'s integer check
  now covers `<<=` and `>>=`.
- `tests/integration/compoundShiftBuild.test.ts` — happy
  path + float-rejection diagnostic.
