# 0.0.1-dev-8 — `Fetch` HTTP client

_Released 2026-05-26_

Mint can now *call* HTTP services in addition to serving them.
Synchronous, no external deps, http:// only (https awaits an
OpenSSL vendor in Phase 5).

## Highlights

```mint
import { Fetch, HttpResponse } from "mint:http";

let r: HttpResponse = Fetch.get("http://example.com/");
println(r.status);                    // 200
println(r.ok);                         // true
println(r.header("content-type"));     // "text/html; charset=utf-8"
println(r.body);                       // the page body
```

Three call-sites cover the common patterns:

| Function | Notes |
| --- | --- |
| `Fetch.get(url)` | plain GET |
| `Fetch.post(url, body, contentType)` | POST, content-type set if non-empty |
| `Fetch.request(method, url, body, contentType)` | any verb |

`HttpResponse` is property-style:

| Accessor | Type |
| --- | --- |
| `status` | int (0 on transport failure) |
| `ok` | bool (status in 200..299 with no error) |
| `error` | string (empty when network call succeeded) |
| `body` | string (raw response payload) |
| `headers` | `HeaderMap` (case-insensitive `[]`) |
| `header(name)` | shortcut returning a single header |

Live-verified: `Fetch.get("http://example.com/")` returns
status 200 with a 540-byte body and `text/html` content-type.

## Implementation

- Synchronous BSD/winsock client (already linked via mint:http's
  `ws2_32`). `#ifdef _WIN32` swaps `closesocket` /
  `WSAStartup` for `close` / nothing.
- `getaddrinfo` + `connect` + `send` + `recv`; loops until peer
  closes. Default user-agent `doublemint/0.0.1`, `Connection:
  close` to keep the parser simple.
- Response parser pulls the status code from the first line,
  splits headers on `\r\n\r\n`, and treats everything after as
  body — no chunked-transfer support yet (server side handles
  it implicitly because `Connection: close` makes upstream
  fall back to identity).
- `HttpResponse` owns a `shared_ptr<ResponseState>` so copies
  are cheap and lifetime survives Mint method-chain semantics.

## Const-correctness on `Http`

While wiring this up the example `server.listen(...)` inside
`Async.spawn(fn(): void => server.listen(...))` failed to
compile because the captured `Http` was const inside the
lambda. Every `Http::*` method now carries a trailing `const`
so server instances can be safely captured by value in
handlers. The `shared_ptr<ServerHolder>` member makes that
const-safe (the holder is what mutates, not the `Http` handle).

## Files touched

- `src/runtime/sources/http.cpp` — fetch.cpp content folded in
  (DNS lookup, raw HTTP/1.1 write, response parser); `Http::*`
  methods marked `const`.
- `src/runtime/headers/http.hpp` — new `HttpResponse` class
  declaration; `Http::*` member declarations marked `const`;
  `WebSocket` already const-correct.
- `src/builtins/mintModules.ts` — registers `HttpResponse`
  class + `Fetch` namespace inside mint:http.
- `examples/builtins/fetch/main.dlm` — walks `Fetch.get`,
  status/body/header inspection, and pretty-prints the first
  200 chars of example.com.
