# 0.0.1-dev-28 — bitwise `& | ^ ~`

_Released 2026-05-26_

Integer bitwise operators land. AND / OR / XOR as binary
ops, NOT as a unary prefix. Shift ops (`<<` / `>>`) are
deferred for the next bump because of the generic-type
tokenizer overlap.

## Highlights

```mint
import { println } from "mint:io";

export function main(): void {
  if ((240 & 15) == 0) { println("and ok"); }
  if ((240 | 15) == 255) { println("or ok"); }
  if ((255 ^ 15) == 240) { println("xor ok"); }
  let mask: int = 255;
  if ((~mask & 255) == 0) { println("not ok"); }
}
```

Precedence (C-style, lowest → highest at this level):

```
||  &&  |  ^  &  == !=  < <= > >=  + -  * / %  unary  postfix
```

So `flag & MASK == 0` parses the C way as
`flag & (MASK == 0)` — paren it as `(flag & MASK) == 0`
to get the human-expected meaning. The emitter wraps any
bitwise binary in parens so a downstream `==` doesn't grab
the operand under `-Wparentheses`.

Type rules:

- `& | ^` require `int` / `int64` (or numeric-literal)
  operands; `DLM4080` otherwise.
- Unary `~` requires the same; `DLM4081` otherwise.
- Result type follows `widerNumericType`, same as the
  other arithmetic ops.

## Files touched

- `src/lexer/token.ts`, `src/lexer/scanner.ts` — `CARET`
  and `TILDE` tokens for `^` and `~`.
- `src/parser/ast.ts` —
  `BinaryExpression.operator` widened with `"&" | "|" |
  "^"`; `UnaryExpression.operator` widened with `"~"`.
- `src/parser/parser.ts` — three new precedence layers
  (`bitwiseOr` → `bitwiseXor` → `bitwiseAnd`) inserted
  between `logicalAnd` and `equality`; unary `~` recognised
  in `unary()`.
- `src/semantic/checker.ts` — extended the integer-only
  check in the binary path to cover `&`, `|`, `^` (sharing
  the `DLM4080` code with `%`); unary `~` integer check
  with `DLM4081`.
- `src/emitter/cppEmitter.ts` — wraps bitwise binaries in
  parens so the C++ side keeps the right precedence.
- `tests/integration/bitwiseBuild.test.ts` — happy path
  plus float-operand and bool-`~` diagnostics.
